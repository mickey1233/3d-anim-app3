# Plan（固定模板）

> 規則：本 Plan 以「可執行」為主，避免空泛描述；所有段落都要能直接轉成工程工作項目與測試點。

## 背景 / 目標

- 背景：本專案為「3D CAD Assembly & Animation Studio v2」重構中產品；README 已列 v2 Core：單頁響應式布局、Parts list/選取/Transform、面基貼合（Mate）、SOP steps + timeline、Command bar（mock router + trace）、VLM 多圖上傳與 mock analyze；並有 v2 WS gateway（typed schema + router + tools registry）。
- Scope Calibration（範圍校準規則）：
  - 覆蓋來源：README v2 Core、既有 `prd/PRD.md`、PRD 附錄 B 的代碼模組清單（`src/v2/*`、`mcp-server/v2/*`、`shared/schema/*`）。
  - P0/DONE 硬規則：不在來源或無合格 Evidence → 不得 P0、不得 DONE（最多 P2 + NOT DONE/IDEA）。
- 文件切分策略（硬規則；分層避免 <8000 chars 與內容完整衝突）：
  - 分層：
    - `prd/PRD.md`：索引 + 決策 + P0 總表（FR/NFR）+ 關鍵契約「摘要」+ 風險/裁決（不放長表與範例 payload）。
    - `prd/MANUAL_CHECKLIST.md`：MT（單一真相）+ 每條 MT 的 `Covers:`/步驟/可觀測預期。
    - `prd/TELEMETRY.md`：NFR 指標表 + TestMethod 細節（含無遙測替代測法）+ 契約範例 payload（如 VLM response 範例）+ trace 限制細節（如需）。
    - `prd/TEST_ANCHORS.json`：僅例外的測試錨點映射（不放規格/需求）。
  - 外移規則：P1/P2 需求清單、契約範例 payload、NFR 測法細節一律外移到固定附檔；`prd/PRD.md` 對每個外移主題必保留 1 段最小摘要 + 連結。
  - 檔名限制：只允許上述三個附檔；不新增其他 `prd/*`。
- 目標（可衡量）：
  - 交付 PRD：產品面章節齊備（Personas/JTBD、Flows、Success Metrics、Glossary、Non-goals），工程面可追溯（P0 FR 皆有 Evidence+AC+TestRef；P0 NFR 皆有可驗收 TestMethod）。
  - 交付可執行驗收來源：`prd/MANUAL_CHECKLIST.md` 為手動驗收單一真相；自動測試以統一錨點可追溯。
  - PRD 版本/日期更新為 2026-02-05；完成 DONE/PARTIAL/NOT DONE 校準與變更紀錄。

## 非目標（Out of Scope）

- 不新增/重做產品功能（只做 PRD/測試/腳本最小對齊；不改 UI/Three/solver 行為）。
- 不替換 mock AI/VLM 為真實供應商整合。
- 不針對 v1 新增需求。
- 不做效能優化實作；僅定義 NFR 與測法。
- 不實作 i18n。
- 不做 session export/import（FR-SOP 明示排除）。

## 需求拆解（使用者故事 / 功能點）

- PRD 產品面（放 `prd/PRD.md`；細節可外移但需摘要+連結）：
  - Personas & JTBD（≥2）、Primary User Flows（≥3）、Success Metrics（≥5）、Glossary、Non-goals。
- ID 規範：
  - FR：`FR-<AREA>-<NNN>`；AC：`AC-<FRID>-<NN>`；MT：`MT-P{0|1|2}-{NN}`；NFR：`NFR-<NNN>`。
  - AREA：`ML/PT/MATE/SOP/CMD/VLM/WS`。
- Success Metrics ↔ NFR（強制對齊）：
  - 每個 Success Metric 引用 0..n `NFR-<NNN>`；若非本輪可測標 `PRODUCT_ONLY`。
  - 每個 P0 NFR 至少被 1 個 Success Metric 引用（M4 validate 檢查存在性）。
- 功能層（FR；P0 只保留總表於 `prd/PRD.md`；P1/P2 細節外移並保留索引）：
  - FR-ML、FR-PT、FR-MATE、FR-SOP（不含 session export/import）、FR-CMD、FR-VLM、FR-WS。
- 非功能層（NFR；P0 摘要在 `prd/PRD.md`，細節與測法在 `prd/TELEMETRY.md`）：
  - P0 NFR 必須可在無遙測下驗收；否則不得列 P0（降級或 Spec-only）。

## 技術方案（架構 / 模組 / 資料流）

- 架構：Frontend（Vite+React+R3F+Zustand）/ Backend（WS gateway+router+tools+VLM mock）/ Shared（zod schema）。
- schemaVersion/Compatibility（摘要在 `prd/PRD.md`，矩陣與測法可在 `prd/TELEMETRY.md`）：
  - Compatibility Rules：`major` 不同 → `VERSION_MISMATCH`；`major` 相同 → capabilities gating（blocked/read-only 觸發條件 + 降級策略）。
  - capabilities defaults（固定集合；找不到 Evidence → 移出 matrix 判定條件並標 NOT DONE）：`toolsRegistry`, `commandRouter`, `vlmAnalyze`, `trace`。
  - FR row 必新增 `DependsOnCaps`：列出依賴 capabilities。
  - 硬規則：若某 capability 找不到 Evidence 而被移出 matrix 判定條件，所有 `DependsOnCaps` 含該 capability 的 P0 FR 必須降級為 P1 或標 PARTIAL（不得維持 P0 DONE）。

## 介面 / 契約（API / 函式簽名 / 事件）

- WS 協議摘要（PRD 只留摘要；範例 payload 外移到 `prd/TELEMETRY.md`）：
  - Hello：`schemaVersion/appVersion/capabilities`
  - Command：`CommandRequest(dryRun?)` / `CommandResult(effects/trace?)`
- Confirm Flow（互斥模式；Evidence 門檻）：
  - 模式 A（Protocol confirm）DONE：`spec:` schema 有欄位 + `proc:` server handler 有處理。
  - 否則模式 B（fallback），且 B1/B2 各需 Evidence。
- Test 錨點（唯一規則，避免覆蓋不可信）：
  - 解析優先順序：先讀 `// @fr ...`；若無則退回 title 含 FR-ID。
  - 多 FR 覆蓋規則（硬規則）：單一測試覆蓋多 FR 必須同一行：`// @fr FR-PT-001 FR-ML-002`，禁止散落多行。
- `prd/TEST_ANCHORS.json`（僅例外；唯一定位）：
  - entry 結構：`{ "frId": "FR-PT-001", "file": "tests/xxx.spec.ts", "testTitle": "..." }`（`testTitle` 必須完整字面量）。
  - validate 行為（M4）：同時命中 `testTitle` 與 `frId` 才算通過；否則輸出錯誤並 `exit 1`。
- Evidence（防亂填）：
  - 偏好：`path#SymbolOrKeyword`；次選：`path:line#Keyword`。
  - Keyword 規則：長度≥3，字元集 `[A-Za-z0-9_./-]`；需可 grep。
  - validate（M4）：若使用 `:line` 但無 `#Keyword` 或 keyword 不合規 → `exit 1`。

## 風險與對策

- PRD 長度 vs 可讀性：分層（PRD 只 P0+索引+摘要；細節外移固定附檔）。
- anchors 映射分裂：`prd/TEST_ANCHORS.json` 僅例外，且用 `testTitle` 唯一定位；validate 檢查嚴格 fail-fast。
- 錨點解析不穩：統一「先 `// @fr` 後 title」，多 FR 同行註解。
- capabilities gating 空話：`DependsOnCaps` +「capability 無 Evidence → 依賴的 P0 FR 強制降級」。
- schema 改動偷渡行為：M3 任何 `shared/schema/*` 變更必同時更新 `tests/v2_contract.spec.ts`，且 PRD 變更紀錄寫 reason；不得要求 server/client 行為改動。
- Evidence keyword 亂填：keyword 規則 + validate 強制。

## 未解分歧與裁決（最多第 5 輪出現）

- **PRD 長度與資訊密度**：折衷採分層；`prd/PRD.md` 僅保留索引+決策+P0總表+契約摘要，細節外移固定附檔並保留最小摘要+連結；理由：同時滿足 <8000 chars 與可交付性。
- **測試錨點雙軌（title vs 註解）**：裁決統一優先 `// @fr`，若不存在才用 title；多 FR 必同一行 `// @fr ...`；理由：可解析且覆蓋關係穩定。
- **TEST_ANCHORS 是否會成第二真相**：裁決僅例外使用，且用 `{frId,file,testTitle}` 唯一定位；validate fail-fast（exit 1）於任何不一致；理由：避免片段搜尋誤命中與映射漂移。
- **Evidence 的 `:line` 使用**：裁決允許但必帶 `#Keyword`（長度≥3、字元集限制），validate 強制；理由：降低行號漂移風險並保留人工定位能力。
- **capabilities defaults 缺 Evidence 對 P0 的影響**：裁決 capability 無 Evidence → 移出 matrix 判定條件；所有依賴它的 P0 FR 必降級或標 PARTIAL；理由：讓相容策略校準對優先級產生硬約束，避免空表。
