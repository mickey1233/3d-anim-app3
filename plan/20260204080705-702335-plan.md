# Plan（固定模板）

> 規則：本 Plan 以「可執行」為主，避免空泛描述；所有段落都要能直接轉成工程工作項目與測試點。

## 背景 / 目標

- 背景：本專案為「3D CAD Assembly & Animation Studio v2」重構中，已具備單頁響應式 UI、零件清單/選取/變換、面基貼合（mate）、SOP 步驟與播放、命令列 + WS router（mock）、VLM 多圖上傳與 mock analyze 等核心能力；需把現況功能與缺口整理成「可交付、可估工、可驗收」的 PRD v1（以 `prd/PRD.md` 草稿為基底）。
- 目標（可衡量）：
  - PRD 交付物（主文 + 必要附錄；主文用相對連結引用附錄）：
    - 主文：`prd/PRD.md`
    - 附錄（必要）：`prd/P0_INDEX.md`、`prd/TRACEABILITY.md`、`prd/VERIFICATION_CHECKLIST.md`、`prd/ID_MAP.md`、`prd/TESTABILITY.md`
    - 附錄（必要，契約/策略）：`prd/WS_CONTRACT.md`、`prd/COMMAND_RUNNER_MATRIX.md`、`prd/TOOLS_SECURITY.md`、`prd/ERROR_CODES.md`、`prd/ERROR_PRESENTATION.md`
    - 附錄（必要，範圍/相容性）：`prd/SUPPORT_MATRIX.md`
    - 附錄（可選，實作參考）：`prd/STORE_HISTORY.md`
  - 「核對基準」必須表格化且可落地（避免空表）：PRD 模板固定使用
    - 欄位：`欄位名 | Required(P0)/Optional(P1) | 說明 | 範例值 | 填入時機(M1模板/M2a/M3/M3.5/M4) | Owner | 更新時機`
    - 每里程碑必填集合（寫死在 PRD；PR 合併前必更新）：  
      - M1：建立模板（不要求實值）  
      - M2a：填 `git commit SHA`、Node/npm、OS、啟動 scripts 名稱/命令（含 WS script）、WS URL/port 優先序草案、驗證資產政策與預設路徑占位  
      - M3：補 Playwright/Browser revision、headless/workers、Playwright browser args/WebGL 後端策略、測試命令模板與「實際採用命令」  
      - M3.5：在同一 commit SHA 上補 Gate B 行號定位與 K/N 統計（含關鍵域覆蓋）  
      - M4：全欄位補齊（含連結檢查、文件大小檢查結果）
  - 追溯表 Gate 分級（避免核心量測被跳過）：
    - Gate A（PRD v1 必達）：P0 100% 具「最短手動驗證步驟」；自動化欄位 100% 非空（`檔名+test title` 或 `N/A + 原因+補測里程碑`）。
    - Gate B（PRD v1.1 必達）：達成行號門檻與關鍵域覆蓋（定義見「驗收標準」），且綁定核對基準 commit SHA。
  - 最小必做 Enablement（僅限啟動 scripts；避免不可重現；M1 必做）：
    - 目標：前端與 WS gateway 都可用 `npm run <script>` 啟動。
    - script 命名與回退：預設 `ws:v2`；若衝突用 `ws:gateway:v2`；**只新增、不改名、不改既有 scripts 行為**；README 保留舊指令仍可用。
  - 時程假設（用於避免 Gate B 無限延後；可在 PRD 記錄為假設並可調整）：
    - 假設 PRD v1（Gate A）目標：`2026-02-07`
    - 假設 PRD v1.1（Gate B）目標：`2026-02-11`（以 M3.5 完成為準）

## 非目標（Out of Scope）

- 不要求在 PRD v1 交付前完成新功能實作；任何功能擴充/改動一律在 PRD 的 P1/P2/Backlog 管理（Plan 不重複維護清單）。
- 不擴寫/重構 legacy v1（僅在 PRD 內註記「仍可用但不再開發」與差異）。
- 不做大規模 UI/架構重構；僅允許 Enablement 最小改動（scripts、testid、tests、Playwright args/CI）且必須可回退；其中只有「啟動 scripts（dev + WS gateway）」為 PRD v1 前置。

## 技術方案（架構 / 模組 / 資料流）

- 架構：
  - Frontend：Vite + React（`src/v2`）+ R3F/Three.js + Zustand + Tailwind UI。
  - Backend：Node/TS WebSocket gateway（`mcp-server/v2`）提供 router/tool registry 與 VLM mock。
  - Shared：Zod schema（`shared/schema`）定義 WS 協議與 payload。
- 模組責任邊界（PRD 必須明確寫出；追溯表需填「模組路徑」）：
  - AppShell/Panel Layout
  - Three Canvas（載入、選取/outline、gizmo/orbit、mating、anchors、step playback）
  - Store/History（snapshot + undo/redo）
  - Command Runner（本地解析/WS fallback、P0 請求生命週期、可觀測事件）
  - WS Client/Gateway（request/response、重連、provider、tool allowlist）
  - VLM（圖片隊列→Analyze→回覆→（P1）落地 steps）
- 追溯表模板（`prd/TRACEABILITY.md` 固定欄位；Gate A/B 依此驗收）：
  - 欄位：`FR-ID | 標題 | Priority | 狀態 | 最短手動驗證步驟 | 自動化（檔名:L行號+test title 或 N/A+原因） | 模組路徑 | Notes`
  - 最短手動步驟粒度：3–7 步，必含「觀測點」與「預期狀態/錯誤碼」；P0 錯誤驗證必含：在 command log 看到 error code。
- Store/History（可驗收行為 vs 實作建議分層）：
  - 需求層（主文）：Undo/Redo 語義、納入 history 的資料、transaction 邊界、chat/trace 非歷史資料。
  - 合併規則（可測）：同一 `requestId` 導致多次 state mutation，對外必須是單一 history entry；Undo 一次回到該 requestId 執行前（對應 AC-HIST-01）。
  - 參考實作（附錄 `prd/STORE_HISTORY.md`）：Implementation default 可調（不寫死在主文）。
- 請求生命週期（P0 最小集合；其餘列 P1；需圖示化）：
  - P0：requestId、亂序對齊、重試策略（二選一鎖定）、AC-REQ-01/02、Mermaid sequence diagram（寫入 `prd/COMMAND_RUNNER_MATRIX.md`）。
  - P1：cancel、多 in-flight 限制。
  - Mermaid sequence diagram 最小範本（含 happy path + retry path；PRD 可擴充不可刪減）：
    ```mermaid
    sequenceDiagram
      autonumber
      participant U as User
      participant FE as Frontend(CommandRunner)
      participant WS as WS Gateway
      participant ST as Store/History
      participant UI as UI(CommandLog/Trace)

      U->>FE: Enter(text)
      FE->>FE: new requestId (uuid)
      FE->>UI: log.append{requestId,text,status=in_flight}

      alt Local-only parsed
        FE->>ST: dispatch(local action, requestId)
        ST-->>FE: historyTxId
        FE->>UI: log.update{requestId,ok=true,historyTxId}
      else WS-only/unknown
        alt WS connected
          FE->>WS: router_execute{requestId,text,attempt?}
          WS-->>FE: response{requestId,ok|error,toolCalls?,trace?}
          FE->>ST: dispatch(toolCalls, requestId)
          ST-->>FE: historyTxId
          FE->>UI: log.update{requestId,ok?,errorCode?,historyTxId,traceRef?}
        else WS disconnected
          FE->>UI: log.update{requestId,ok=false,errorCode=ERR_NOT_CONNECTED,retryable=true,inputPreserved=true}
        end
      end

      U->>FE: Retry(same entry)
      FE->>FE: retry policy (attempt++ or parentRequestId)
      FE->>WS: router_execute{requestId/parent,attempt,text}
      WS-->>FE: response{...}
      FE->>ST: dispatch(..., requestId/parent)
      ST-->>FE: historyTxId
      FE->>UI: log.update by requestId/parent
    ```
- Tool allowlist（P0 最小清單；寫入 `prd/TOOLS_SECURITY.md`）：
  - allowlist 初始來源：後端 registry 為唯一事實來源（優先引用 `mcp-server/v2/tools/registry.ts`；若實際不同需記錄路徑與 export）。
  - P0 最小：列出所有 state-mutation tools（以 registry 實際工具為準），每個工具需標副作用分級、Undo 期待、`historyTxId` 可觀測。
- Error Presentation Matrix（P0 最低一致性；寫入 `prd/ERROR_PRESENTATION.md`）：
  - P0：所有錯誤至少在 command log 顯示 error code；文案固定語言；追溯表以 error code 為主。

## 介面 / 契約（API / 函式簽名 / 事件）

- WS 契約（PRD 必須同時提供：schema 路徑+export、欄位對照表、JSON 範例、錯誤碼策略、請求生命週期）：
  - 連線預設：`ws://localhost:3011`（最終以核對基準記錄的 scripts 啟動結果為準）。
  - WS URL/Port 優先序（Plan 建議草案；PRD 需以程式碼驗證後鎖定）：  
    1) `import.meta.env.VITE_WS_URL` → 2) default `ws://localhost:3011` → 3) 其他覆寫（若有）需明示 key/優先級
  - 核心 message（至少 `router_execute`、`vlm_analyze`）：欄位對照表 + 成功/失敗 JSON 範例。
- VLM Analyze（P0）：
  - 允許 mime types：`image/png|image/jpeg|image/webp`
  - `images[].id` 生成責任（可測、可追溯）：前端於「加入隊列」時生成 UUIDv4，存入 Zustand `vlmImages`（或同義 slice）；刪除移除，id 不復用；重試沿用既有 id；排序不改 id。
- 請求生命週期（P0 契約）：
  - 重試策略二選一（P0 必鎖定）：A) `requestId+attempt` 或 B) `parentRequestId`
  - 亂序回覆：以 requestId（或 parent/attempt 規則）對齊，不得以到達順序覆寫狀態
  - AC-REQ-01/02 必須進 PRD，並在追溯表對應到測試（或 `N/A + 補測里程碑`）
- Store/History 可測 AC：
  - AC-HIST-01：同一 requestId 造成多次 state mutation，Undo 一次回到該 requestId 執行前狀態；command log 可觀測 `historyTxId` 與 requestId 對齊。
- 錯誤呈現一致性（P0 契約）：
  - error code 必顯示且至少出現在 command log；追溯表手動步驟以此為主要觀測點。

  - 對策：P0 最低一致性寫死：所有錯誤至少在 command log 顯示 error code；追溯表手動步驟必含此觀測。

## 驗收標準（與 PRD 連動）

- 定義：
  - N = P0 總數（`prd/P0_INDEX.md` 列數）。
  - K = P0 中「自動化欄位含行號 `:L\d+`」的數量。
- Gate A（PRD v1 必達）：
  - `prd/P0_INDEX.md` 逐條列舉 P0（不得用區間）。
  - `prd/TRACEABILITY.md`：P0 100% 手動步驟 + 自動化欄位 100% 非空。
- Gate B（PRD v1.1 必達）：
  - `K ≥ min(10, ceil(0.6N))` 且 FR-WS/FR-VLM/FR-MATE/FR-SOP 各 ≥1 條有行號；行號綁定核對基準 commit SHA。
- 啟動可重現（最小 Enablement）：
  - 前端與 WS gateway 均可用 `npm run <script>` 啟動（`npm run dev` + `npm run ws:v2` 或 fallback）。
- 錯誤可驗收（最低一致性）：
  - 所有錯誤至少出現在 command log 並顯示 error code。
- 核對基準可落地：
  - 核對基準表格含 Required/Optional/Owner/更新時機，且每里程碑 Required 欄位已填。
- 文件大小上限（可維護性）：
  - `prd/PRD.md` 建議 ≤ 30KB；單一附錄建議 ≤ 20KB；`wc -c` 結果記錄於核對基準。

## 未解分歧與裁決（最多第 5 輪出現）

- 本 Plan 輪次：第 5 輪（最大輪次；需在本輪完成折衷與凍結）
- 仍有歧見點與折衷（以可測性/契約一致性/最小改動為準）：
  - hot-swap FR-ID 是否固定為 `FR-ML-09`
    - 歧見：固定可方便引用；但可能撞草稿既有編號。
    - 折衷：`FR-ML-09` 僅作候選；M2a 盤點後落定 `FR-ML-0X`（下一可用連號）；以 `prd/ID_MAP.md` 作唯一映射來源。
    - 理由：避免破壞連號與既有追溯，同時保留可引用的落定流程。
  - Layout/WS/VLM 是否應新建 FR-UI/FR-WS/FR-VLM 群組
    - 歧見：新群組清晰；但可能與草稿同義 FR 重複。
    - 折衷：允許新增群組，但必須先盤點草稿是否有同義 ID；若有同義必沿用；若無同義才新建；任何新舊對應一律寫入 `prd/ID_MAP.md`。
    - 理由：確保契約一致與統計不失真。
  - Gate B（行號門檻）是否會拖慢交付
    - 歧見：Gate B 成本偏高；但若無行號可定位性不足。
    - 折衷：PRD v1 以 Gate A 為硬門檻；PRD v1.1 以 Gate B 為硬門檻，並前移到 M3.5（不壓在封存 M4）。
    - 理由：兼顧「先可交付」與「後可定位」兩種品質要求，避免最後一刻阻塞。
  - WS gateway script 命名（`ws:v2` vs `mcp:v2`）與衝突處理
    - 歧見：命名偏好不同、可能已有同名。
    - 折衷：預設 `ws:v2`；衝突 fallback `ws:gateway:v2`；只新增不改名；README 保留舊指令；核對基準記錄實際採用 script 名稱。
    - 理由：最小改動、可回退、易於驗收。
  - 文件大小檢查要不要做成 `npm run prd:check`
    - 歧見：用 script 一致；但非核心且可能增加 enablement。
    - 折衷：M4 以「核對基準記錄 `wc -c` 結果」為必達；`npm run prd:check` 列為可選 enablement（不阻塞 Gate A）。
    - 理由：確保可重現與一致入口，同時避免過度擴 scope。
