# Plan（固定模板）

> 規則：本 Plan 以「可執行」為主，避免空泛描述；所有段落都要能直接轉成工程工作項目與測試點。

## 背景 / 目標

- 背景：
  - 專案為「3D CAD Assembly & Animation Studio v2」：前端單頁（React + R3F/Three + Zustand），後端為 v2 WebSocket gateway（router + VLM mock），並保留 legacy v1（`?legacy=1`）作對照。
  - repo 已有 `prd/PRD.md`（草稿），但需基於「現有功能/測試/契約參考（`shared/schema`）」整理成「可驗收、可追溯、可維護」的 PRD v1：補齊 P0 的 AC（Given/When/Then + Pass 條件）、錯誤狀態（可機器斷言的 `code/retryable/uiMessageKey`）、契約範例（samples + schemaRef）、以及需求↔測試追溯（traceability）。
  - 假設（不問問題，先寫入 PRD 並可被驗證）：
    - v2 前端 dev：`http://localhost:5173`；legacy：`?legacy=1`；WS gateway：`ws://localhost:3011`（以 README.md 為準）。
    - `shared/schema` 在 M0–M2 視為「建議存在的契約參考」（不阻塞）；自 M3（契約 samples 上線）起視為「必要」（會阻塞）。
- 目標（可衡量）：
  - P0 凍結且可驗收：PRD 內有 `P0 Scope Table`，表內每個 P0 ID 皆具備「AC/Errors/Manual verify」三段且符合最小語法（可觀察 Pass 條件），可被 `validate-prd --p0-completeness` 機器檢查。
  - 追溯可機器檢查：以可配置測試掃描規則（`scripts/prd-check.config.json`）為準，`prd/traceability.json`（versioned schema）覆蓋全部 discovered tests 並通過 `validate-prd --traceability`（含 P0 orphan 防呆）。
  - 生成/檢查流程穩定：`prd:gen` 固定產生/正規化 generated 檔（commands inventory + traceability normalize）；`prd:check:*` 先跑 `prd:gen:check` 再跑 validator（分階段 checksets），避免 generated 檔過期造成誤 fail/誤 pass。

## 非目標（Out of Scope）

- 不進行大幅產品功能開發或 UX 改版（核心 3D 求解/播放/主要面板結構不改）。
- 僅允許最小行為修正的邊界（為了可測/可回退/契約一致性）：
  - 僅限：mock provider deterministic（斷言以 `code/structure/order` 為主）、錯誤格式（taxonomy + 結構 + uiMessageKey）、重連/timeout/backoff 參數一致性。
  - 必須可回退：單獨 commit 可 `git revert`；若需並存新舊行為，僅允許 `MOCK_V2_COMPAT=1` 類型環境旗標切換，並在 Decisions 記 removal criteria/target milestone。
- 不引入外部解析/驗證依賴：所有 scripts 僅用 Node built-ins；不引入 yargs/commander；不做 zod/TS runtime schema 欄位驗證（P1 研究項）。

## 需求拆解（使用者故事 / 功能點）

- PRD 文件交付物（必做）
  - 作為讀者，我能在 PRD 看到：P0 凍結清單、每個 P0 的 AC/錯誤狀態/手動驗證（含 Pass 條件）、以及 P0 外的 P1/P2 roadmap。
  - 作為工程，我能用結構化檔（`prd/commands-inventory.json`、`prd/traceability.json`）把「commands/測試」與需求對齊並降低漂移。
  - 作為維護者，我能用 `prd:check:*` 一鍵檢查 PRD：章節、ID、顯式錨點、P0 完整性（Given/When/Then + Pass）、錯誤碼格式/唯一性、追溯覆蓋、samples（JSON parse + schemaRef 路徑規範 + schemaRef 檔案存在性）。
- 產品功能層（以現況為準，P0 由 Scope Table 凍結）
  - 模型載入/視覺化、選取/變換、Mate、SOP、Command/Router 降級、VLM 僅候選護欄、NFR（分層 gate：量測可產生但不硬 fail；契約類可硬 fail）。

## 技術方案（架構 / 模組 / 資料流）

- 架構（PRD 固化為文字 + 圖）
  - 前端：Vite + React + R3F/Three + Zustand。
  - 後端：WS gateway（router + VLM mock）。
- 工具/檔案策略（以「可測、可落地、可回退」為主）
  - Validator（Node>=18/ESM、僅 built-ins）：採固定 PRD 標記與顯式錨點，避免通用 Markdown parser 的脆弱性。
  - 可配置測試掃描：`scripts/prd-check.config.json` 定義 `testsRootDirs[]` + `excludeDirNames[]`，避免 repo 重構測試目錄即阻塞或誤掃描。
  - generated 檔案納入標準流程：`prd:gen`（write）與 `prd:gen:check`（check），由 `prd:check:*` 統一調用。
  - 分階段 gate：提供 `prd:check:m1`、`prd:check:m2`、`prd:check:full`，避免把後期能力（schema-samples/shared-schema-required/traceability lint strict）過早變成日常阻塞。

## 介面 / 契約（API / 函式簽名 / 事件）

- `scripts/prd-check.config.json`（測試掃描/契約檢查配置）
  - schema（v1）：
    - `version: 1`
    - `testsRootDirs: string[]`（預設 `["tests"]`；repo root 相對路徑）
    - `excludeDirNames?: string[]`（預設 `["node_modules","dist","playwright-report",".git"]`；discover/traceability 共用）
    - `protocolVersionField?: string`（預設 `"protocolVersion"`；避免硬編欄位名）
    - `schemaRefAllowedPrefixes?: string[]`（預設 `["shared/schema/"]`）
    - `requireSharedSchema?: boolean`（預設 `false`；M3 會改為 `true`）
    - `strictTraceabilityLint?: boolean`（預設 `false`；M3 視收斂情況決定是否改為 `true`）
- PRD 文件可解析規範（供 `validate-prd`）
  - 僅支援：ATX headings（`#`）、fenced code blocks（```）、固定 HTML 註解標記（`<!-- prd:... -->`）。
  - 不依賴 renderer slug：所有需求以顯式錨點 `<a id="FR-V2-..."></a>`；內部連結只能用 `#FR-V2-.../#NFR-V2-...`。
- P0 凍結與可驗收格式
  - `P0 Scope Table` 標記：`<!-- prd:p0-scope:start -->`…`<!-- prd:p0-scope:end -->`
  - 欄位固定：`id | title | type(FR/NFR) | source | verification(auto/manual-only) | notes`
  - `source` 必填非空（validator 只檢查非空）；格式建議：`README` / `tests/<file>.spec.ts` / `shared/schema/...` / `src/...` / `mcp-server/...` / `Decision:<id>`
  - 若 `verification=manual-only`：`notes` 必須含 reason（非空）。
- 每個 P0 需求的固定區塊（`--p0-completeness` 需檢查）
  - 顯式錨點：`<a id="FR-V2-..."></a>` / `<a id="NFR-V2-..."></a>`
  - AC 區塊：`<!-- prd:ac:start -->`…`<!-- prd:ac:end -->`
    - 最小語法（硬規則）：
      - 必須包含 `Given:`、`When:`、`Then:` 三行，且冒號後內容非空
      - `Then:` 行必須包含 `Pass:`（可觀察結果 token）
  - Errors 區塊：`<!-- prd:errors:start -->`…`<!-- prd:errors:end -->`
    - 最小語法（擇一，硬規則）：
      - 至少一筆錯誤條目行，包含 `code=`、`retryable=`、`uiMessageKey=`（順序不限），或
      - 明確宣告無錯誤狀態且附理由：`errors: none (reason=<non-empty>)`
  - Manual verify 區塊：`<!-- prd:manual-verify:start -->`…`<!-- prd:manual-verify:end -->`
    - 最小語法（硬規則）：至少 2 個步驟（需同時包含 `1.` 與 `2.`），且至少 1 行包含 `Pass:` 或 `Pass 條件：`
- Protocol versioning/compatibility（必須裁決為單一策略 + sample 一致性）
  - 章節標記擇一（且只能擇一）：
    - `<!-- prd:protocol-versioning:explicit -->`（payload 帶版本欄位）
    - `<!-- prd:protocol-versioning:implicit -->`（payload 不帶版本欄位，靠演進規則）
  - 必須提供 3 條演進範例（additive / unknown field / breaking）。
  - 版本欄位名由 `scripts/prd-check.config.json:protocolVersionField` 決定（預設 `protocolVersion`）。
  - 一致性（validator 規則）：
    - 若 `explicit`：所有 `router_execute.*` 的 sample JSON 必須包含版本欄位（由 config 指定）
    - 若 `implicit`：sample 出現版本欄位只出 WARN（允許存在但不得宣稱依賴它）
- Error taxonomy 與 UI mapping（避免文案耦合）
  - 統一錯誤物件欄位（P0 契約）：
    - `code: string`
    - `retryable: boolean`
    - `uiMessageKey: string`
    - `details?: Record<string, unknown>`（用途：傳遞額外上下文，例如 `{ requestId, originalError }`；不得作為穩定斷言依賴）
  - `code` 命名規則（寫入 PRD，並由 `--error-codes` 檢查）：`ERR_V2_<AREA>_<REASON>`（全大寫底線）。
  - Error mapping table（P0）：`code -> uiMessageKey -> defaultMessage -> retryable`
    - `defaultMessage` 僅文件範例，不作契約/測試斷言；validator 對 `defaultMessage` 變更僅 WARN。
- `--error-codes`（錯誤碼格式/唯一性）
  - 掃描範圍：PRD 的 Errors 區塊內 `code=` 條目（與可選 mapping 區段）。
  - 檢查：格式符合規則、全文唯一（duplicate → ERROR）。
- `shared/schema` 參考與檢查（不讀內容）
  - `--shared-schema`（依 config 決定嚴格度；單一規則、可預期）：
    - `requireSharedSchema=false`（M0–M2）：`shared/schema` 不存在或為空 → WARN（不阻塞），且 PRD Sources of Truth 必須記 GAP
    - `requireSharedSchema=true`（M3+）：`shared/schema` 不存在或為空 → ERROR（阻塞）
- Commands inventory（可重複生成 + best-effort 掃描 + 權威覆寫）
  - 設定檔：`scripts/commands-entry-files.json`
    - schema（v1）：
      - `version: 1`
      - `entryFiles: string[]`
      - `manualEntries?: { commandName, handler, sourceFile, isLocal, notes? }[]`（權威覆寫/補漏）
      - `ignorePatterns?: string[]`（best-effort，降低誤判）
  - 生成檔：`prd/commands-inventory.json`
    - schema（v1）：`version: 1`、`entryFiles: string[]`、`entries: [...]`
  - `scripts/gen-commands-inventory.mjs`（built-ins、無 AST）
    - `--write`：寫入 generated 檔
    - `--check`：不寫入；若輸出與檔案不一致則 exit 1
  - `--commands-inventory`（validator）檢查：
    - schema 欄位存在
    - `entryFiles` 與 `entries[].sourceFile` 皆存在於檔案系統（缺失→ERROR）
    - `entries.length > 0`（空陣列→ERROR；若掃描抓不到，必須用 `manualEntries` 補到非空）
- Traceability（結構化、版本化、分階段 lint）
  - 檔案：`prd/traceability.json`（schema v1：`version:1`、`entries`）
  - `type` 判定標準（寫入 PRD/README）：
    - `ui-flow`：主要驗證 UI 流程/DOM
    - `contract`：主要驗證契約（錯誤結構/grammar/WS fallback）
    - `perf`：主要量測數值
  - `--traceability` 硬檢查：
    - 依 `testsRootDirs` 遞迴掃描所有 `*.spec.ts`（套用 `excludeDirNames`）；每個測試必須有 entry 或 excluded（含 reason）
    - `coversIds` 必須存在於 PRD 顯式錨點（或 P0 IDs）
    - P0 IDs 不可成孤兒（覆蓋或 manual-only+reason）
  - lint（`primaryAsserts/type` 一致性）項目定義：
    - `type=ui-flow` 且 `primaryAsserts` 包含 `error-taxonomy/command-grammar/mock-determinism` → lint 違規
    - `type=perf` 但 `primaryAsserts` 未包含 `perf-metrics` → lint 違規
  - WARN→ERROR 收斂標準（可機器判定 + 可回退）：
    - `strictTraceabilityLint=false`（預設/M2b-1）：lint 違規一律 WARN
    - `strictTraceabilityLint=true`（可在 M3 啟用）：lint 違規一律 ERROR
    - M3 啟用條件（寫入 Decisions 並可驗證）：lint 違規數量降到 0 才允許切換為 `true`；若無法降到 0，必須在 Decisions 記錄 exception + removal criteria，維持 `false`
- Traceability normalize（納入 `prd:gen`，必須「不改語意」）
  - `scripts/normalize-traceability.mjs` 規格：
    - 只允許排序/格式正規化：
      - `entries` 依 `testPath` 升冪排序
      - 每筆內 `coversIds/primaryAsserts/incidentalAsserts` 依字母升冪排序
    - 禁止改寫：`notes`、`excludedReason`、字串內容本體
    - JSON 輸出格式固定（2 spaces + 結尾換行）
    - `--check` 以「canonical 輸出」與現有檔案做 byte-for-byte 比對；不同則 exit 1
- PRD JSON 範例（schema samples）標記與 `schemaRef`
  - 標記：code fence info 必須包含 `schemaRef=`（路徑字串）
    - 例：```json prd-sample router_execute.request schemaRef=shared/schema/...
  - `schemaRef` 路徑規範（硬規則）：
    - 必須為 repo-root 相對路徑（不得為絕對路徑、不得含 `..`、不得含 Windows drive like `C:`、不得含 `\`）
    - 必須以 config `schemaRefAllowedPrefixes` 其中之一為前綴（預設 `shared/schema/`）
  - `--schema-samples` 檢查：
    - ERROR：JSON 可 parse
    - ERROR：`schemaRef` 符合路徑規範且指向檔案存在
    - ERROR：若 sample 含 `error` 物件，必須具備 `code/retryable/uiMessageKey`
    - 並依 versioning 規則檢查版本欄位（explicit 必須含 config 指定欄位）
- `validate-prd` CLI 契約（argv 解析 + 穩定錯誤碼）
  - 解析策略：使用 `process.argv` 手動解析；支援 `--checks=a,b,c` 與 `--checkName`（等價加入 checks 集合）。
  - 輸出格式（固定，便於 CI 解析）：
    - `[ERROR][<CHECK>][<E_CODE>] <message>`
    - `[WARN][<CHECK>][<W_CODE>] <message>`
    - `[SUMMARY] errors=<n> warnings=<m>`
  - 核心錯誤碼（至少需穩定，供 CI/統計；可擴充但不可改義）：
    - `E_CFG_INVALID`
    - `E_DISCOVER_NO_TESTS`
    - `E_P0_SCOPE_MISSING`
    - `E_P0_MISSING_BLOCK`
    - `E_P0_AC_MISSING_GWT`
    - `E_P0_AC_MISSING_PASS`
    - `E_P0_ERRORS_NONE_NO_REASON`
    - `E_ERR_CODE_DUPLICATE`
    - `E_CMDINV_EMPTY_ENTRIES`
    - `E_TRACE_MISSING_ENTRY`
    - `E_SAMPLE_SCHEMA_REF_INVALID`
  - 核心警告碼（至少需穩定；不影響 exit code）：
    - `W_SHARED_SCHEMA_MISSING_OR_EMPTY`（`requireSharedSchema=false` 時）
    - `W_VERSION_FIELD_PRESENT_UNDER_IMPLICIT`
    - `W_TRACE_LINT_VIOLATION`
    - `W_ERROR_MAPPING_DEFAULT_MESSAGE_NON_CONTRACTUAL`
- `prd:gen/prd:check:*`（generated 檔案流程 + 分階段 checksets）
  - `prd:gen`（固定包含）：
    - `gen-commands-inventory --write`
    - `normalize-traceability --write`
  - `prd:gen:check`（固定包含）：
    - `gen-commands-inventory --check`
    - `normalize-traceability --check`
  - `prd:check:m1`（M1 gate；不包含後期阻塞項）：先 `prd:gen:check`，再跑 validator：
    - `discover-tests,sections,toc,glossary,decisions,changelog,p0-scope,commands-inventory`
  - `prd:check:m2`（M2 gate）：先 `prd:gen:check`，再跑 validator：
    - `discover-tests,sections,toc,glossary,decisions,changelog,p0-scope,protocol-versioning,ids,links,p0-completeness,error-codes,commands-inventory,traceability`
  - `prd:check:full`（M3+ gate）：先 `prd:gen:check`，再跑 validator：
    - `discover-tests,sections,toc,glossary,decisions,changelog,p0-scope,protocol-versioning,ids,links,p0-completeness,error-codes,commands-inventory,traceability,schema-samples,shared-schema`
  - `prd:check`：固定等同 `prd:check:full`

## 風險與對策

- shared/schema 從「可選」到「必要」的切換會卡住：用 `requireSharedSchema=false→true` 明確落在 M3，並在 M0–M2 以 WARN + PRD GAP 提前揭露風險。
- schemaRef 路徑耦合導致跨平台脆弱：制定 repo-root 相對路徑 + 允許前綴 + 禁止 `..`/絕對路徑/反斜線；validator 同步硬檢查。
- generated 檔案過期造成誤 fail/誤 pass：以 `prd:gen:check` 納入所有 `prd:check:*` 前置步驟，CI/local 一致。
- P0 completeness 形式通過但不可驗收：把 AC/Errors/Manual verify 最小語法升級為可觀察 token（`Then: ... Pass:`、manual 至少 2 步 + Pass 條件；errors:none 必須 reason）。
- commands inventory 掃描誤差導致 entries 為空：將 `entries.length>0` 視為硬條件；掃描不足以 `manualEntries` 補正（並在 PRD 註記 best-effort 限制）。
- traceability lint 過早強約束導致隱性重寫測試：用 `strictTraceabilityLint` 開關與「違規=0 才可升級」的客觀條件；否則 Decisions 記 exception+退場條件。
- 最小修正越界：限定範圍（mock/錯誤格式/重連參數），單獨 `[prd-compat]` commit 可 revert，且至少 1 人 review；必要時 `MOCK_V2_COMPAT=1` 並定義退場條件。

## 里程碑與切分（可分 PR/commit）

> 每個里程碑都要列「DoD + 檔案清單（預計會改哪些路徑） + 最短瀏覽器驗證步驟 + 自動化測試點」

- Size 對照（估時基準）：
  - S：≤0.5 天
  - M：1–2 天
  - L：3–5 天
- 依賴/並行：
  - 依賴鏈：M0a → M0b → M1 → M2a-1 → M2a-2（阻塞）
  - 可並行：M2b-1 可在 M2a-1 後啟動並與 M2a-2 並行；M2b-2 可與 M2b-1 並行
  - M3 需等待：M2a-2 + M2b-1

- M0a（size: S）:
  - DoD：
    - 新增 `scripts/prd-check.config.json`（v1）：包含 `testsRootDirs/excludeDirNames/protocolVersionField/schemaRefAllowedPrefixes/requireSharedSchema/strictTraceabilityLint`（先設 `requireSharedSchema=false`、`strictTraceabilityLint=false`）。
    - 新增 `scripts/validate-prd.mjs` 最小版本：
      - argv 手動解析（`--checks=` / `--checkName`）
      - 固定輸出格式 `[ERROR][CHECK][E_CODE]` / `[WARN][CHECK][W_CODE]` / `[SUMMARY]`
      - 實作並通過 checks：`discover-tests`（依 config 掃描，count>0 才 pass）、`shared-schema`（在 `requireSharedSchema=false` 時缺失/為空 → WARN）
  - Files：
    - `scripts/prd-check.config.json`
    - `scripts/validate-prd.mjs`
  - Browser verification：
    - 無（此里程碑只建工具骨架）。
  - Automated tests：
    - `node scripts/validate-prd.mjs --checks=discover-tests,shared-schema`

- M0b（size: S）:
  - DoD：
    - 新增 `scripts/commands-entry-files.json`（v1）並填入初始 `entryFiles`（以 README/PRD 草稿路徑為準；若實際不存在需在 PR 說明並替換為現況路徑）：
      - `src/v2/ui/CommandBar/useCommandRunner.ts`
      - `mcp-server/v2/tools/registry.ts`
      - `mcp-server/v2/index.ts`
      - `mcp-server/v2/wsGateway.ts`
    - 新增 `scripts/gen-commands-inventory.mjs`：
      - 讀取 `scripts/commands-entry-files.json`

## 未解分歧與裁決（最多第 5 輪出現）

- 角色/Owner 缺位（TL/PM 皆為 `<TBD>`）
  - 折衷：若 TL/PM 未指派，暫由 PR Author 代行決策；所有代行決議必須寫入 PRD 的 Decisions（含日期/理由/回退方式/退場條件）。
  - 理由：避免因 Owner 缺位導致整條交付鏈卡死，且 Decisions 可追溯可回退。
- Protocol versioning 策略（explicit vs implicit）
  - 折衷：以「現況成本最小」為準——若現況 payload 已有版本欄位則選 explicit；否則選 implicit 並在 Decisions 記錄未來導入 explicit 的遷移條件（例如 v3 message type）。
  - 理由：避免「寫文件 → 被迫改碼」的高風險路徑，同時保留可演進性。
- traceability lint 是否升級為 strict（WARN→ERROR）
  - 折衷：僅在 lint 違規數量降為 0 時才允許 `strictTraceabilityLint=true`；否則維持 WARN，並以 Decisions 記錄 exception + removal criteria（何時/如何把違規清零）。
  - 理由：把「不重疊」的理想狀態做成可驗證目標，但不強迫在現有測試現況下做計畫外重構。
- `MOCK_V2_COMPAT=1` 是否引入
  - 折衷：只有在 P0 契約（`code/structure/order`、錯誤結構、重連參數）不一致且阻塞 `prd:check:m2`/`prd:check:full` 時才引入；一旦引入必須寫 removal criteria/target milestone。
  - 理由：保留回退能力與 CI 穩定性，但避免長期堆積行為分歧與技術債。
