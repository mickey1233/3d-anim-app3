# PRD（固定模板）

> 規則：本 PRD 以「可測」為主；每個 Must 需求都要能對應到驗收標準與測試計畫。

## 目標與成功指標

- 目標：
  - 在瀏覽器中提供可用的 v2「3D CAD 組裝 / SOP 工作流」：載入模型 → 選取/變換 → 面基貼合（Top/Bottom）→ 新增 SOP step（含 timeline）→ Command bar 執行本地指令（含 trace）→ VLM（mock）多圖上傳/重排/刪除/Analyze 並顯示結構化結果。
  - 提供「可追溯」需求與測試：所有 Must（P0）需求皆對應到明確驗收標準（Given/When/Then）與測試計畫（unit/integration），可用 `npm test` 重現（`FR-WS-001` 為條件式 Must，需另外啟用）。
  - v2 作為預設 UI；v1 僅保留相容入口（`?legacy=1`），不新增需求。
- 成功指標（可量化）：
  - SM-01（10 分鐘 P0 Flow 完成率 / 手動基線）：
    - 定義：以本節 `MT-P0-01..MT-P0-10` 為單一真相；測試者從開啟 v2 起計時，**10 分鐘內完成 ≥8/10** 視為成功；以 10 次試跑統計成功率 ≥80%。
    - 誰執行：QA（優先）；無 QA 時由開發者輪值。
    - 何時執行：每次 Release Candidate（RC）前必做一次；平時可選每週一次。
    - 記錄位置（硬規則）：在 RC 發佈用 issue（或 release checklist issue）的 comment 以固定欄位記錄並保留最近 10 次。
    - 記錄欄位（固定）：`日期(YYYY-MM-DD) | 執行者 | 成功(Y/N) | 完成項數(0-10) | 失敗項目(MT IDs) | 備註`
    - 失敗處理：若成功率 <80% 或單次流程中任何 Must 需求失敗 → **阻擋 release**，並開 issue 標註 `P0-regression`。
    - MT 與 AC 的裁決規則（硬規則，避免漂移）：
      - MT：只負責「操作步驟 + 觀測點」。
      - AC：只負責「可測判定口徑」。
      - 若同一行為敘述衝突：**以 AC 為準**。
      - 任何 AC 變更必同步檢查並更新受影響 MT（覆蓋：FR-APP/ML/PT/MATE/SOP/CMD/VLM）。
    - MT（每條 ≤5 步 + 1 個可觀測預期；括號內為對應 FR/AC）：
      - MT-P0-01（FR-APP-001 / AC-FR-APP-001-01）v2 入口與版面可操作  
        1) 啟動前端（`npm run dev`）  
        2) 開 `http://localhost:5173`  
        3) 確認未帶 `?legacy=1` 且顯示 v2 介面  
        4) 依序切換到 Parts/Mate/SOP/Command/VLM 面板（或入口）  
        - 預期：面板入口可操作；不需水平捲動才能點到主要入口/輸入框。
      - MT-P0-02（FR-ML-001 / AC-FR-ML-001-01）載入 GLB/GLTF  
        1) 開啟模型載入入口  
        2) 選一個有效 GLB/GLTF  
        3) 等待載入完成  
        - 預期：載入期間有指示；完成後畫布可見模型（非空畫面），且可進行一次零件選取。
      - MT-P0-03（FR-PT-001 / AC-FR-PT-001-01）Canvas 選取有 outline  
        1) 在畫布點選任一零件 mesh  
        - 預期：該零件出現 outline/高亮。
      - MT-P0-04（FR-PT-001 / AC-FR-PT-001-01）Parts list 同步高亮 + partId 可取得  
        1) 先完成 MT-P0-03 的選取  
        2) 觀察左側 Parts list  
        - 預期：對應項目高亮；可讀到穩定 `partId`（`data-part-id`）。
      - MT-P0-05（FR-PT-002 / AC-FR-PT-002-01）Transform 拖曳造成變化  
        1) 選取零件  
        2) 拖曳 Transform gizmo 任一軸  
        - 預期：零件 position/rotation 數值可觀測變化。
      - MT-P0-06（FR-PT-002 / AC-FR-PT-002-01 + AC-FR-PT-002-02）拖曳期間 orbit 停用、結束恢復  
        1) 非拖曳狀態右鍵拖曳畫布一次（orbit）  
        2) 拖曳 gizmo 同時右鍵拖曳畫布  
        3) 結束拖曳後再右鍵拖曳畫布  
        - 預期：步驟(1)(3)視角可旋轉；步驟(2)視角不旋轉。
      - MT-P0-07（FR-MATE-001 / AC-FR-MATE-001-01A + AC-FR-MATE-001-02）Mate Top/Bottom 預覽 + Apply  
        1) 在 Mate 面板選 source/target 零件  
        2) 選 Top/Bottom face（至少 top/bottom）  
        3) 進入 preview 狀態  
        4) 按 Apply  
        - 預期：面板顯示 preview 與 anchor method 文案（若 fallback 顯示）；Apply 後 transform 有變化且未報錯；source=target 時 Apply 禁用/報錯且不變更。
      - MT-P0-08（FR-SOP-001 / AC-FR-SOP-001-01 + AC-FR-SOP-001-02）新增 step + timeline 更新  
        1) 在 SOP 面板新增 step  
        2) 點選該 step  
        - 預期：steps panel 出現新 step 且有可見標識（id/title/placeholder）；timeline 更新並反映選取狀態。
      - MT-P0-09（FR-CMD-001 / AC-FR-CMD-001-01 + AC-FR-CMD-001-02 + AC-FR-CMD-001-03）Command bar 本地指令 + trace  
        1) 輸入 `grid off` 送出  
        2) 輸入 `nudge x 0.01` 送出  
        3) 輸入 `grid abc` 送出  
        - 預期：前兩次成功且 trace 各新增一列 ok；第三次顯示錯誤且 trace 新增一列 error；trace 為追加、不覆寫。
      - MT-P0-10（FR-VLM-001 / AC-FR-VLM-001-01 + AC-FR-VLM-001-02）VLM 多圖操作 + Analyze 不改變 3D/SOP  
        1) 上傳至少 2 張圖片  
        2) 重排順序並刪除其中 1 張  
        3) Analyze（mock）  
        - 預期：顯示結構化結果；Analyze 前後零件 transform 不變、SOP steps 數量不變；空圖集 Analyze 會提示且不送出；結果可觀測到 detections 的 `imageIndex`。
  - SM-02（P0 自動化覆蓋 / 排除條件式 Must）：
    - 定義：`npm test` 覆蓋所有 **非條件式 Must** 的主流程斷言（每個 Must FR ≥1 斷言）且綠燈；覆蓋依據以「Coverage Matrix（P0 Must）」驗收（見 `## 測試計畫`）。
    - 例外規則（硬規則）：Must 不允許以 `skip` 隱含處理缺少可測錨點；必須先滿足 P0 可測錨點要求（`data-testid` / `data-part-id` / 可觀測 `imageIndex`），否則該需求不得維持 Must。
  - SM-03（Transform 可用性）：拖曳期間 orbit 停用、結束恢復（E2E：以 quaternion 角距離判斷 + 時間窗，見 AC-FR-PT-002-01/02）。
  - SM-04（可觀測性 / trace 關聯鍵）：
    - 優先使用 `requestId`（若 UI 或 schema 提供）。
    - 若無 requestId：使用 UI trace **序號（整數遞增）** 作替代關聯鍵（優先序號避免時間格式 flake）。
  - SM-05（VLM 操作完整性）：Analyze（mock）前後 **3D 零件 transform 與 SOP steps 數量不變**（以穩定 `partId` 重定位 + 浮點容差比較，見 AC-FR-VLM-001-01）。
  - SM-06（可靠性底線 / 條件式）：
    - 目標：`npm test` 執行期間 `pageerror`/`unhandledrejection` = 0。
    - 機制（可執行，見 NFR-002）：以 `prd/nfr_baseline.json` 決定 soft/hard；不存在時預設 soft（不阻擋但輸出報告）；檔案存在但欄位缺失/型別錯誤則設定錯誤直接 fail。

## 使用者情境 / 用例

- UC-01 製程工程師：匯入 GLB/GLTF 組裝模型，從 Parts list 選取零件，使用 Transform gizmo 做初步對位，再用 Mate（Top/Bottom）精準貼合，將貼合結果記錄成 SOP steps，於 timeline 逐步切換驗證序列。
- UC-02 技術訓練人員：上傳多張組裝照片到 VLM 面板、調整順序、刪除不需要的圖，按 Analyze（mock）取得結構化 steps 建議，人工比對後回到 3D 場景逐步操作並記錄 SOP。
- UC-03 設計驗證工程師：用 Command bar 輸入本地指令（grid/nudge/reset），快速迭代並透過 trace 理解系統採取的行為與錯誤原因。

## 功能需求（Must / Should / Could）

### Must
- FR-APP-001（P0）單頁響應式 v2 介面：
  - v2 為預設入口；v1 以 `?legacy=1` 存取。
  - 面板可折疊；在 `1024x768` 視窗下可完成 P0 流程；「無水平捲動」以元素必在 viewport 內且可互動為準（見 AC-FR-APP-001-01）。
  - 可測錨點要求（P0，**不可缺**）：以下元素需提供穩定 selector（優先 `data-testid`）以支援 E2E：
    - `parts-panel`, `mate-entry`, `sop-entry`, `timeline`, `command-input`, `vlm-entry`
  - 若現況缺少 testid 的裁決（硬規則）：本輪需補齊上述 `data-testid`；否則 `SM-02` 無法成立且視為 P0 未達（不接受改用不穩定 selector 代替）。
- FR-ML-001（P0）模型載入與基本視覺化（GLB/GLTF）：
  - 載入 GLB/GLTF（含可選 Draco）；載入時顯示載入狀態；成功後相機 auto-fit。
  - P0 自動化驗收口徑：僅驗「載入完成可選取 + 非空畫面」（見 AC-FR-ML-001-01），避免幾何/視椎判定 flake；auto-fit 視覺品質由手動 MT 保底。
  - 大檔案提示：當檔案 `size > 50MB` 顯示警告但不阻止載入（測試用 mock size 驗收）。
  - Draco（可選）規則：若載入的是 draco 壓縮資產且不支援，必須顯示包含 `Draco` 關鍵字的錯誤訊息（避免 silent fail）。
- FR-PT-001（P0）Parts list + 選取：
  - Canvas 點選 mesh 可選取並以 outline 高亮；左側 Parts list 同步高亮同一零件。
  - 可測錨點要求（P0，**不可缺**）：Parts list item 必須提供穩定 `partId`，且 P0 自動化一律使用 `data-part-id`（不依賴「可複製文字」流程）：
    - `data-part-id="<partId>"`
- FR-PT-002（P0）Transform 操作（含 Orbit 停用/恢復）：
  - Transform gizmo 可拖曳選取零件，拖曳造成零件 transform 數值改變。
  - 拖曳期間 Orbit 必須停用、結束恢復；以「角距離公式」驗證並含時間窗與明確 orbit 觸發方式（右鍵拖曳，見 AC-FR-PT-002-01/02）。
- FR-MATE-001（P0）面基貼合（Top/Bottom 最低可用）：
  - 使用 Mate 面板選 source/target 零件與 face（至少 Top/Bottom）。
  - P0 preview 驗收以「Mate panel 顯示 preview 狀態 + anchor method 文案 + fallback 文案（若有）」為準；3D marker 可視化定位屬 Should（見 FR-MATE-101）。
  - Apply 後 source 零件 transform 更新且未回報錯誤（integration 可測）。
  - 幾何誤差計算以 Geometry Validator 規格做單元測試（不要求 solver 內部可直接取 face mesh），見 AC-FR-MATE-001-01B。
- FR-SOP-001（P0）SOP steps + timeline（add/select）：
  - 可新增 step、可選取 step；timeline bar 會更新並反映目前選取。
  - steps panel 需能顯示 step 的最小可見標識（id/title/placeholder，見 AC-FR-SOP-001-02）。
  - 不包含 session export/import。
- FR-CMD-001（P0）Command bar + trace（本地指令一致回報）：
  - 本地指令最小清單（P0）：`grid on/off`、`nudge <axis> <delta>`、`reset part <name>`、`reset all`。
  - 每個本地指令的預期行為（P0，可測）：
    - `grid on`：顯示網格
    - `grid off`：隱藏網格
    - `nudge <axis> <delta>`：將「目前選取零件」沿指定軸位移 `delta`（axis 只允許 `x|y|z`；delta 為 number，可正/負/0）
    - `reset part <name>`：將指定零件還原初始 transform
    - `reset all`：所有零件還原初始 transform
  - name 匹配規則（P0，避免不可預期）：
    - 來源：Parts list 顯示名（UI 顯示的 part label）
    - 匹配：精確匹配（大小寫敏感）
    - 找不到：error（訊息需包含 `not found` 或「找不到」關鍵字其一）
    - 多筆同名：error（訊息需包含 `ambiguous` 或「不唯一」關鍵字其一）
  - P0 trace 最小格式（可測）：每次 command 後 trace 列表「新增一列」（不可覆寫），且該列至少顯示：`序號(整數遞增)` + `status(ok|error)` + `message`（可空）。
- FR-VLM-001（P0）VLM 面板（mock）：
  - 多圖上傳/重排/刪除；Analyze（mock）回傳結構化結果並顯示（steps/detections/partMappingCandidates）。
  - Analyze 前後不得自動變更 3D 零件 transform 或 SOP steps（以穩定 `partId` 重定位 + 浮點容差比較驗收；不得以 skip 取代可測錨點）。
  - 空圖集 Analyze：顯示提示且不送出請求（P0）。
  - `detections[].imageIndex` 語義（P0）：從 0 開始，對應 VLM 面板「重排後」當前顯示順序索引。
  - 可測輸出錨點要求（P0，**不可缺**）：Analyze 結果區必須能觀測到 `detections[].imageIndex`（至少擇一：在 detections 列表顯示 `imageIndex`；或提供 raw JSON 展開/複製視窗）。
  - mock 最小回傳規格（為了可測與契約一致）：對於輸入 `N` 張圖片，mock 至少回傳 `N` 筆 detections，使得 `imageIndex` 覆蓋 `0..N-1`（每張至少 1 筆 detection）。
- FR-WS-001（P0，條件式 Must）v2 WS gateway 基本互通：
  - 僅在「WS gateway 已啟動」前提下驗收。
  - 前端可連線到 `ws://localhost:3011` 完成 hello 握手（schemaVersion/capabilities）。
  - 能送出 `CommandRequest` 並收到 `CommandResult`（至少 `ok` 或 `error` 一種路徑），且有可理解 `message` 或 trace 事件。

## 非功能需求（效能 / 安全 / 相容 / 可觀測性）

- NFR-001（可用性）所有 Must 操作均有明確 UI 回饋（選取 outline、timeline 更新、trace 追加列、VLM 結果顯示、錯誤訊息）。
- NFR-002（可靠性底線，條件式 Must；可執行機制）：
  - 測試收集：Playwright 收集 `pageerror`/`unhandledrejection`，輸出到報告（含前 3 筆摘要）。
  - 基線檔：`prd/nfr_baseline.json`
  - baseline 檔 schema（最小）：
    - `mode: "soft" | "hard"`
    - `allowedPageErrorCount: number`
    - `allowedUnhandledRejectionCount: number`
  - 預設行為（硬規則，避免不確定）：
    - 檔案不存在：預設 `mode="soft"`；`allowed*` 視為極大值（不阻擋）；但必輸出統計與摘要。
    - 檔案存在但欄位缺失/型別不符：視為設定錯誤，測試輸出明確 error 並 fail（exit 1）。
- NFR-003（相容性）以 Chromium/Chrome 桌面為主；v1 仍可透過 `?legacy=1` 進入（不要求功能一致）。
- NFR-004（可觀測性）trace row 最小規格（schema-agnostic）：
  - 追加語義：每次 command 後 trace 必「新增一列」，不得覆寫既有列。
  - 序號：整數遞增，且在頁面存活期間不重置（驗收以序號為優先；timestamp 可有可無）。
  - 每列至少可見：`序號` + `status(ok|error)` + `message`（可空）。
  - 關聯鍵：優先 `requestId`；若無，使用序號。
  - timestamp/序號裁決：自動化驗收以「序號存在且遞增」為優先（避免時間格式/時區造成 flake）。
- NFR-006（載入體驗 / 大檔案提示，P0 可測）：門檻 50MB；`file.size > 50MB` 時顯示警告但不阻止載入（mock size 觸發）。

## 邊界案例與錯誤處理

- 模型載入：
  - 不支援/損毀檔案：顯示可理解錯誤訊息；不應卡死 UI。
  - 大檔案（>50MB）：顯示警告 + 載入指示（NFR-006）。
  - Draco 不支援：顯示包含 `Draco` 關鍵字的錯誤訊息。
- 選取/Transform：
  - 未選取零件時操作：顯示「未選取」狀態，避免 silent no-op。
  - 拖曳中 Orbit 干擾：拖曳期間 orbit 不生效；拖曳結束恢復（AC 覆蓋）。
- Mate：
  - source 與 target 為同一零件：禁用 Apply 並顯示「不可將零件與自身貼合」；不得套用 transform。
  - 無法求解：顯示錯誤原因；不得套用 transform（no-op）。
  - 判定條件（最小可測定義）：若 source/target face 法向量平行且同向，且無 twist/翻面策略時，視為不可求解（回報 error，不套用 transform）。
- SOP：
  - 新增 step 後 timeline 不更新：視為 P0 失敗。
- Command/WS：
  - WS 未啟動：顯示紅色 banner「後端連線失敗，部分功能不可用」；不得崩潰。
  - schema 驗證失敗：拒絕套用 effects（若有），顯示「回應格式錯誤」，trace 仍追加 error 事件（若 trace UI 可用）。
- VLM：
  - 空圖集 Analyze：提示且不送出。
  - 單張圖片 Analyze：正常處理。
  - 重排後 `imageIndex`：必對應重排後 UI 順序（AC 覆蓋）。

## 介面規格（API / 輸入輸出 / 狀態碼）

- 前端啟動：
  - v2 UI：`npm run dev` → `http://localhost:5173`
  - v1 UI：`http://localhost:5173/?legacy=1`
- WS gateway（README 指定）：
  - 啟動：`npx tsx mcp-server/v2/index.ts`
  - 預設：`ws://localhost:3011`
- WebSocket 訊息（以 `shared/schema` 的 zod schema 為準；此處為最小摘要）：
  - Hello：`ClientHello/ServerHello`（schemaVersion/capabilities）
  - Command：`CommandRequest` → `CommandResult`
  - VLM：`VlmAnalyzeRequest(images[])` → `VlmAnalyzeResponse(steps[],detections[],partMappingCandidates[])`
- WS 失敗 UI（Must，可測）：
  - WS 連線失敗或握手失敗時：在 Command bar 下方顯示紅色 banner：「後端連線失敗，部分功能不可用」。

## 資料結構 / Schema（若需要）

> 本節為「可測」而寫；實作以 `shared/schema/*` 為唯一真相。若 schema 未包含此處欄位，則該欄位視為 Should/Could 或標 NOT DONE。

- Step（SOP，P0：add/select）：
  - `id: string`
  - `title?: string`（若無 title，steps panel 需顯示 placeholder）
  - `snapshotRef?: string`
  - `createdAt?: number`
- Mate（P0：Top/Bottom + preview + apply）：
  - `sourcePartId: string`, `targetPartId: string`
  - `sourceFace: "top"|"bottom"|...`, `targetFace: "top"|"bottom"|...`
  - `anchorMethod: string`
  - `offsets?: { x:number, y:number, z:number }`
- Geometry Validator（P0 單元可測規格；固定 world space，避免口徑分裂）：
  - 座標空間：全部為 world space。
  - normal：必須為 unit vector（長度=1；若輸入非 unit，先 normalize）。
  - angle（deg）：normal 夾角（取最小夾角）。
  - distance（點到平面距離）：
    - `distance = abs(dot((sourceAnchor - targetAnchor), targetNormal))`
  - 門檻：
    - `angleEpsDeg = 5`
    - `distanceEps = max(0.001, 1e-5 * bboxDiagonalWorld)`
- VLM Analyze Response（P0：顯示即可）：
  - `steps[]`: `{ title: string, description?: string, confidence?: number }`
  - `detections[]`: `{ label: string, bbox: [number,number,number,number], score: number, imageIndex: number }`
  - `partMappingCandidates[]`: `{ detectionId: string, partId: string, score: number, suggestionType: string, rationale?: string }`
  - `detections[].imageIndex`：從 0 開始，對應 VLM 面板「重排後」當前顯示順序索引。

## 未解分歧與裁決（最多第 5 輪出現）

- MT vs AC 的裁決規則（避免漂移）：裁決 **MT 只負責操作步驟與觀測點，AC 只負責可測判定口徑**；若兩者敘述衝突，以 AC 為準；任何 AC 變更需檢查並同步受影響 MT（APP/ML/PT/MATE/SOP/CMD/VLM）。
- `data-testid` / `data-part-id` 前置：裁決其為 P0 可測錨點且不可缺；若現況未實作，需先補齊再宣稱 P0 Must 達標（不接受改用不穩定 selector 代替）。
- VLM `imageIndex` 可測性：裁決 Analyze 結果必能觀測 `imageIndex`（detections 列表顯示或 raw JSON）；mock 需保證 `imageIndex` 覆蓋 `0..N-1` 以便驗收；否則視 P0 不達標。
- `auto-fit` 的自動化口徑：裁決 P0 自動化不做幾何精確判定，改以「載入完成 + 可選取 + 非空畫面」驗收；精確 auto-fit 視覺品質由手動 MT 保底。
- `reset part <name>` 匹配策略：裁決採「Parts list 顯示名精確匹配」；找不到/不唯一必 error + trace 追加列，避免不可預期。
